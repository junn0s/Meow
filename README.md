# 냥포차: 달빛 아래 야식당

> **Meow Night Diner** — 비 오는 밤, 작은 포장마차를 달빛 아래 인기 야식당으로 키우는 2D 픽셀아트 경영 시뮬레이션

`냥포차: 달빛 아래 야식당`은 한 판에 약 5~8분 동안 손님의 주문을 받고, 음식을 조리·서빙해 번 돈으로 가게를 성장시키는 웹 게임입니다. 초반에는 치즈 고양이 사장이 직접 움직이지만, 셰프와 서버를 고용하면 조리와 서빙이 자동화됩니다. 10단계 성장을 마치고 달빛 간판과 평점 5점을 달성하면 클리어합니다.

## 핵심 특징

- Phaser 3, TypeScript, Vite 기반의 설치 없는 브라우저 게임
- `480 × 270` 논리 좌표계를 `960 × 540` 실제 버퍼에 렌더링하고, 글자는 2배 해상도·픽셀아트는 NEAREST로 분리 처리
- 손님 입장 → 좌석 배정 → 주문 → 조리 → 서빙 → 식사 → 결제 → 퇴장의 상태 기반 흐름
- 어묵, 떡볶이, 붕어빵 메뉴와 데이터 기반 10단계 업그레이드
- 셰프 고양이의 자동 조리와 서버 고양이의 자동 서빙
- `localStorage` 기반 진행 저장 및 이어하기
- 메뉴·일시 정지·오프라인 시간을 제외한 활성 플레이 시간 저장
- 외부 이미지·음원 없이 코드로 생성한 픽셀아트와 Web Audio API 효과음
- 키보드 이동·상호작용과 마우스 업그레이드 구매 지원
- 이동·주문·성장 규칙을 안내하는 3단계 시작 튜토리얼

## 실행 방법

### 요구 환경

- Node.js 20 이상 권장
- npm 10 이상 권장
- Web Audio API와 `localStorage`를 지원하는 최신 데스크톱 브라우저

### 개발 서버

```bash
npm install
npm run dev
```

터미널에 표시된 로컬 주소를 브라우저에서 엽니다. Web Audio 정책상 효과음은 첫 키 입력 또는 클릭 이후 활성화될 수 있습니다.

### 프로덕션 빌드 확인

```bash
npm run typecheck
npm run build
npm run preview
```

빌드 산출물은 `dist/`에 생성됩니다. 제출 전에는 개발 서버뿐 아니라 `npm run preview`에서도 새 게임, 저장 복원, 클리어까지 확인해야 합니다.

2026-07-15 최종 통합본에서 `npm run typecheck`와 `npm run build`를 실행해 모두 통과했습니다. 프로덕션 빌드는 잠금 파일 기준 Vite 6.4.3으로 30개 모듈을 처리하고 `dist/`를 생성했습니다.

## 조작법

| 입력 | 기능 |
| --- | --- |
| `WASD` 또는 방향키 | 치즈 고양이 이동 |
| `Space` | 가까운 손님의 주문 받기, 완성 음식 들기, 음식 서빙 |
| 마우스 클릭 | 업그레이드 선택·구매, 메뉴 버튼 조작 |
| `Esc` | 일시 정지 또는 설정 화면 열기·닫기 |
| `M` | 효과음 켜기·끄기 |

플레이어는 한 번에 음식 하나만 들 수 있습니다. 상호작용 대상 가까이 가면 안내가 나타나며, 손님이 남긴 돈은 가까이 가면 수거됩니다. 직원이 해금되기 전까지는 주문과 서빙을 직접 처리해야 합니다.

## 게임 진행

1. 손님이 입구로 들어와 빈 좌석을 배정받습니다.
2. 손님 가까이에서 `Space`를 눌러 주문을 받습니다.
3. 주문은 해금된 조리대의 대기열에 들어가 일정 시간 후 완성됩니다.
4. 조리대 가까이에서 `Space`를 눌러 음식을 들고, 해당 손님에게 서빙합니다.
5. 손님은 식사를 마친 뒤 메뉴판 가격에 `밤손님 영업 보너스 ×3`가 붙은 정산액과 대기 시간에 따른 팁을 지불합니다.
6. 번 돈으로 다음 업그레이드를 구매해 메뉴, 좌석, 직원과 시설을 확장합니다.
7. 달빛 간판을 구매하고 가게 평점 5점을 달성하면 클리어 화면으로 이동합니다.

### 메뉴

| 메뉴 | 기본 조리 시간 | 메뉴판 가격 | 해금 시점 |
| --- | ---: | ---: | --- |
| 어묵 | 3초 | 10냥 | 처음부터 |
| 떡볶이 | 5초 | 25냥 | 떡볶이 조리대 구매 후 |
| 붕어빵 | 7초 | 45냥 | 붕어빵 기계 구매 후 |

표의 가격은 메뉴판 기준입니다. 해커톤 MVP에서 5~8분 안에 성장 흐름을 경험할 수 있도록 실제 결제의 기본 정산액은 `메뉴판 가격 × 3`(밤손님 영업 보너스)으로 계산합니다. 팁 10% 또는 30%도 이 보너스 적용 정산액을 기준으로 반올림하며, 이 규칙은 게임 튜토리얼과 결제 표시에서 함께 안내합니다.

### 10단계 성장

| 단계 | 시설·업그레이드 | 비용 | 효과 |
| ---: | --- | ---: | --- |
| 1 | 어묵 조리대 | 무료 | 어묵 판매 시작 |
| 2 | 추가 좌석 | 50냥 | 최대 손님 수 2명 증가 |
| 3 | 빠른 냄비 | 100냥 | 모든 음식 조리 시간 30% 단축 |
| 4 | 떡볶이 조리대 | 200냥 | 떡볶이 판매 시작 |
| 5 | 셰프 고양이 | 350냥 | 조리 자동화 |
| 6 | 추가 테이블 | 500냥 | 최대 손님 수 2명 증가 |
| 7 | 붕어빵 기계 | 700냥 | 붕어빵 판매 시작 |
| 8 | 서버 고양이 | 900냥 | 서빙 자동화 |
| 9 | 네온 간판 | 1,200냥 | 손님 방문 간격 30% 감소 |
| 10 | 달빛 간판 | 2,000냥 | 최종 가게 완성 및 클리어 조건 해금 |

## 저장 데이터

돈, 구매한 업그레이드, 방문 손님 수, 평점, 튜토리얼 완료 여부, 음소거 설정과 활성 플레이 시간은 브라우저의 `localStorage`에 저장됩니다. 같은 브라우저와 같은 배포 주소에서 다시 접속하면 저장 데이터를 복원합니다.

활성 플레이 시간 `elapsedMs`는 실제 게임 장면이 진행되는 동안에만 증가합니다. 메뉴·튜토리얼·일시 정지·오프라인 시간은 결과 화면의 영업 시간에 포함되지 않습니다. 브라우저 저장소와 메모리 fallback에 모두 저장본이 있으면 각 데이터의 유효성을 확인한 뒤 `lastSavedAt`이 더 최신인 값을 선택합니다. 새 게임을 시작해도 기존 음소거 설정은 보존됩니다.

- 시크릿 모드, 사이트 데이터 삭제, 저장 공간 제한에 따라 데이터가 사라질 수 있습니다.
- 배포 경로가 달라지면 브라우저가 서로 다른 출처로 인식하여 기존 저장을 읽지 못할 수 있습니다.
- 저장 구조가 바뀔 때를 대비해 저장 데이터에는 버전을 두고, 읽을 수 없는 값은 안전한 초기 상태로 대체해야 합니다.

## 기술 구성

```text
브라우저 입력
   ↓
Phaser Scene ── HUD·업그레이드 패널
   ↓
손님·좌석·주문·조리·경제·업그레이드 시스템
   ↓
데이터 정의(메뉴·업그레이드·손님) ── 저장 시스템(localStorage)
   ↓
코드 생성 텍스처(Phaser Graphics/Canvas) + Web Audio API 효과음
```

주요 책임은 다음처럼 분리합니다.

- **Scene**: 화면 전환, 게임 루프 연결, 입력 전달
- **Entity**: 플레이어·손님·직원·테이블·조리대의 현재 상태와 표시
- **System**: 좌석 예약, 주문 대기열, 조리 타이머, 결제, 업그레이드 효과 같은 게임 규칙
- **Data**: 메뉴와 10단계 업그레이드의 가격·효과를 코드와 분리해 선언
- **UI**: HUD, 상호작용 안내, 업그레이드 패널, 시작·클리어 화면
- **Save**: 저장 데이터 직렬화, 버전 확인, 최신본 선택, 활성 시간·설정 복원과 초기화
- **Audio/Art**: 외부 파일을 내려받지 않고 런타임에 그래픽과 효과음을 생성

세부 설계는 [게임 설명서](docs/game-description.md), AI 협업 기록은 [AI 활용 보고서](docs/ai-usage-report.md), 에셋 정책은 [에셋 및 라이선스](docs/asset-licenses.md)를 참고하세요.

## GitHub Pages 배포

1. 저장소에 코드를 푸시합니다.
2. `npm run build`가 성공하고 `dist/index.html`이 만들어지는지 확인합니다.
3. 이 프로젝트의 프로덕션 빌드는 Vite `base: "./"`를 사용합니다. 생성된 `dist/index.html`의 스크립트·스타일 경로가 상대 경로인지 확인합니다.
4. GitHub 저장소의 **Settings → Pages**에서 배포 방식을 **GitHub Actions**로 선택합니다.
5. Actions 워크플로에서 의존성을 설치하고 빌드한 뒤 `dist/`를 Pages artifact로 업로드·배포합니다.
6. 배포 URL을 시크릿 창에서 열어 직접 새 게임과 저장 복원을 확인합니다.

저장소가 `https://<사용자>.github.io/<저장소>/`에 배포되므로 로컬 루트(`/`)를 가정한 에셋 경로를 사용하면 안 됩니다. 현재의 상대 `base` 설정과 Pages 워크플로가 실제 배포 URL에서도 동작하는지 반드시 확인하세요.
