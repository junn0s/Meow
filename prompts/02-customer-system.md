# 02. 손님·좌석 시스템 프롬프트

## 목적

손님이 입장해 좌석을 예약하고 주문을 기다린 뒤 결제·퇴장할 수 있는 안정적인 상태 흐름을 만든다.

## AI에 전달할 프롬프트 전문

```text
현재 Phaser 3 + TypeScript 게임 "냥포차: 달빛 아래 야식당"에 손님과 좌석 시스템을 추가해 줘.

먼저 기존 파일과 책임 구조를 확인하고, 사용자 코드나 이미 동작하는 프로젝트 설정을 불필요하게 다시 작성하지 마. TypeScript strict와 any 금지 원칙을 유지해 줘.

손님 상태는 다음 순서를 기본으로 한다.

ENTERING
→ WAITING_FOR_SEAT
→ MOVING_TO_SEAT
→ ORDERING
→ WAITING_FOR_FOOD
→ EATING
→ PAYING
→ LEAVING

필수 요구사항:

1. Customer 엔티티와 CustomerState 타입 또는 enum을 만든다.
2. Table 또는 Seat 엔티티는 고유 id, 좌표, occupied/reserved 상태, 배정 손님 id를 추적한다.
3. CustomerSystem은 일정한 생성 간격으로 손님을 만든다.
4. 첫 손님은 실제 플레이에서 15초 이내에 등장해야 한다.
5. 손님은 입구에서 빈 좌석을 검색한다.
6. 좌석을 선택한 순간 즉시 예약한다. 도착한 뒤에만 점유 처리해서 두 손님이 같은 좌석을 고르는 버그가 생기면 안 된다.
7. 빈 좌석이 없으면 입구의 겹치지 않는 대기 위치에서 기다리고, 좌석이 비면 도착 순서대로 다시 배정한다.
8. 손님은 목표 지점까지 프레임 시간(delta)을 반영해 이동한다. 프레임률에 따라 속도가 달라지면 안 된다.
9. 좌석 도착 후 ORDERING 상태로 바꾸고 주문 아이콘을 표시할 수 있는 인터페이스를 제공한다.
10. 이번 단계에서 주문 시스템이 아직 없어도 다음 단계가 WAITING_FOR_FOOD로 전환할 수 있는 명시적 메서드 또는 이벤트 경계를 만든다.
11. 식사 시간 종료 후 PAYING을 거쳐 돈을 남길 위치를 제공하고 LEAVING으로 전환한다.
12. 출구에 도달한 손님을 화면과 시스템 컬렉션에서 제거한다.
13. 정상 퇴장, 강제 제거, 장면 종료 모든 경우에 좌석 예약과 점유를 정확히 해제한다.
14. 동일한 손님을 두 번 제거하거나 좌석을 두 번 해제해도 다른 손님의 좌석 상태가 손상되지 않게 한다.
15. Customer 엔티티는 자기 표시·이동·현재 상태를 관리하고, 좌석 검색과 전체 생성 정책은 시스템이 관리한다.
16. Phaser 물리 엔진은 꼭 필요한 부분에만 사용하고 단순 목표 이동에 과도하게 의존하지 않는다.
17. 토끼, 강아지, 햄스터, 너구리 네 손님 종류를 데이터로 정의한다.
18. 외형은 외부 스프라이트 없이 코드 생성 픽셀 텍스처로 구분한다.
19. 손님별 선호 메뉴 id를 둘 수 있지만, 주문 시스템이 없는 현재 단계에서 가격이나 해금 규칙을 중복 구현하지 않는다.
20. 인내심 최대값과 현재값을 갖고 ORDERING/WAITING_FOR_FOOD 동안 감소시킬 수 있게 한다.
21. MVP에서는 인내심이 0이 되어도 즉시 주문을 취소하지 않고, 이후 팁에 영향을 주는 값으로 유지한다.
22. 손님 머리 위에 상태 또는 인내심을 표시할 수 있는 UI hook을 만든다.
23. 추가 좌석 업그레이드가 나중에 런타임으로 좌석을 등록할 수 있게 SeatSystem API를 제공한다.
24. 직원 자동화가 손님 상태를 안전하게 읽고 전환할 수 있게 public API를 명시한다.

상태 전환 안전 규칙:

1. 허용되지 않은 상태 전환은 무시하거나 개발 환경에서 명확한 오류를 낸다.
2. 좌석 없는 손님을 MOVING_TO_SEAT로 바꾸지 않는다.
3. WAITING_FOR_FOOD인 손님에게만 음식 전달이 가능하도록 다음 시스템이 검사할 수 있게 한다.
4. 퇴장 중인 손님에게 새 좌석이나 주문을 배정하지 않는다.
5. 장면 pause 시 인내심과 식사 타이머도 함께 pause되어야 한다.
6. 장면 shutdown 시 생성 timer와 손님 이벤트를 모두 해제한다.

디버그와 검증:

1. 개발 모드에서만 손님 id, 상태, 좌석 id, 목표 좌표를 볼 수 있는 선택적 디버그 표시를 제공한다.
2. 손님 생성 간격을 테스트 설정으로 0.2초까지 줄일 수 있게 하되 프로덕션 기본값과 분리한다.
3. 손님 20명을 빠르게 생성해 한 좌석에 둘 이상 예약되지 않는지 검사할 수 있게 한다.
4. 좌석 0개, 1개, 추가 좌석 구매 후 상태를 각각 확인한다.
5. 타입 검사와 npm run build를 실제로 실행하고 결과를 보고한다.

구현 후 생성·수정 파일, 책임, 확인한 상태 전환, 알려진 한계, 실행한 명령과 실제 결과를 알려 줘. 실행하지 않은 검증은 "검증 예정"으로 남겨 줘.
```

## 기대 검증

- 20명 연속 생성에도 같은 좌석 id를 동시에 가진 손님이 없다.
- 좌석이 비면 가장 먼저 기다린 손님이 이동한다.
- 장면 재시작 후 이전 손님과 생성 타이머가 남지 않는다.
- 추가 좌석을 런타임 등록해도 기존 좌석 id와 충돌하지 않는다.
