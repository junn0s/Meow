# 04. 업그레이드·저장·클리어 프롬프트

## 목적

돈을 소비해 가게가 시각적·기능적으로 성장하는 10단계 업그레이드, 안전한 저장 복원, 최종 클리어를 구현한다.

## AI에 전달할 프롬프트 전문

```text
현재 "냥포차: 달빛 아래 야식당" 프로젝트에 데이터 기반 10단계 업그레이드, localStorage 저장, 클리어 흐름을 추가해 줘.

기존 경제·손님·주문·조리·직원 시스템을 먼저 확인하고 그 API를 사용해. 업그레이드 UI가 직접 돈이나 게임 객체를 조작하지 않게 하고, TypeScript strict와 any 금지를 유지해.

업그레이드 데이터는 다음 순서와 값을 단일 기준으로 사용한다.

1. 어묵 조리대 / 무료 / 어묵 판매 시작
2. 추가 좌석 / 50냥 / 좌석 2개 추가
3. 빠른 냄비 / 100냥 / 모든 음식 조리 시간 30% 단축
4. 떡볶이 조리대 / 200냥 / 떡볶이 판매 시작
5. 셰프 고양이 / 350냥 / 접수된 주문 조리 자동화
6. 추가 테이블 / 500냥 / 좌석 2개 추가
7. 붕어빵 기계 / 700냥 / 붕어빵 판매 시작
8. 서버 고양이 / 900냥 / 완성 음식 서빙 자동화
9. 네온 간판 / 1,200냥 / 손님 방문 간격 30% 감소
10. 달빛 간판 / 2,000냥 / 최종 가게 완성, 클리어 조건 활성화

업그레이드 규칙:

1. 업그레이드는 id, 이름, 설명, 비용, 선행 id, 효과 종류, 효과 값을 갖는 데이터로 정의한다.
2. 모든 단계는 바로 전 단계를 선행 조건으로 갖고 순서대로 한 번만 구매한다.
3. 1단계 어묵 조리대는 새 게임 초기 상태에 구매 완료로 적용하거나 비용 0 구매로 한 번 처리한다. 두 방식이 섞여 중복 효과가 생기지 않게 명확히 선택한다.
4. 현재 구매 가능한 다음 단계만 강조하고, 미래 단계는 잠금 이유를 표시한다.
5. 잔액이 부족하면 구매 버튼을 비활성화하고 필요한 금액을 보여 준다.
6. EconomySystem이 잔액, 선행 조건, 중복 구매 여부를 최종 검증하고 성공할 때만 한 번 차감한다.
7. 더블 클릭이나 같은 프레임의 중복 이벤트로 두 번 결제되지 않게 한다.
8. 구매 성공·실패를 명시적 결과 타입으로 반환해 UI가 한국어 메시지를 표시하게 한다.
9. 구매 성공 시 Web Audio API로 합성한 효과음을 재생하고 외부 음원은 사용하지 않는다.

효과 적용:

1. 좌석 업그레이드는 SeatSystem API로 고유 id 좌석 두 개를 실제 등록하고 화면에 코드 생성 테이블/의자를 표시한다.
2. 빠른 냄비는 모든 메뉴에 전역 0.7 조리 시간 배율을 한 번 적용한다. 저장 복원 때 0.49로 중복 적용되면 안 된다. 이는 원 기획의 어묵 전용 효과를 MVP에서 단순화한 확정 변경이다.
3. 메뉴 해금은 OrderSystem이 읽는 해금 집합을 바꾸고 기존 데이터와 중복 정의하지 않는다.
4. 셰프와 서버 업그레이드는 기존 직원 자동화 시스템을 활성화한다.
5. 네온 간판은 프로덕션 기본 손님 생성 간격에 0.7 배율을 한 번 적용한다.
6. 달빛 간판은 코드 생성 간판 외형을 최종 단계로 바꾸고 클리어 판정을 활성화한다.
7. 모든 효과는 구매 직후 화면과 실제 규칙에 함께 반영된다.

저장 요구사항:

1. 서버나 로그인 없이 localStorage를 사용한다.
2. 저장 데이터에는 version, money, purchasedUpgradeIds, customerCount, rating, lastSavedAt을 포함한다.
3. 필요하면 총 플레이 시간이나 설정 값을 선택적으로 추가하되 런타임 객체와 timer 자체를 직렬화하지 않는다.
4. 돈 획득, 업그레이드 구매, 평점 변화, 장면 종료처럼 중요한 시점에 저장한다.
5. 너무 잦은 프레임별 쓰기는 하지 않는다.
6. JSON parse 실패, null, 배열 대신 다른 타입, 음수 돈, NaN, 알려지지 않은 업그레이드 id를 검증한다.
7. 읽을 수 없는 데이터는 게임을 중단시키지 말고 안전한 새 게임 기본값으로 복구한다.
8. 저장 스키마 버전이 다르면 명시적인 migration을 하거나 안전하게 초기화한다.
9. 복원 순서는 기본 월드 생성 후 업그레이드 효과 재구성, 마지막에 돈·평점·통계를 적용하는 식으로 결정적이어야 한다.
10. 복원 중에는 비용을 다시 차감하거나 구매 효과음을 반복 재생하지 않는다.
11. 새 게임/저장 초기화 버튼은 사용자 확인 후 해당 게임 키만 삭제한다.
12. 저장 키에는 프로젝트 고유 이름과 스키마 버전 체계를 사용해 다른 앱과 충돌하지 않게 한다.

클리어 요구사항:

1. 달빛 간판 구매와 가게 평점 5점을 모두 만족해야 클리어한다.
2. 둘 중 하나만 충족한 상태에서는 무엇이 남았는지 HUD에 표시한다.
3. 조건이 충족되면 클리어 처리를 한 번만 실행한다.
4. 손님 생성과 입력, 조리·식사 타이머를 안전하게 멈춘다.
5. ResultScene에 최종 돈, 방문 손님 수, 평점, 플레이 시간을 전달한다.
6. ResultScene은 다시 시작과 저장된 가게 보기 경로를 제공할 수 있게 한다.
7. 클리어 효과음과 달빛 연출은 코드와 Web Audio로 생성한다.

밸런스 목표:

1. 첫 업그레이드는 정상 플레이 1분 안에 구매할 수 있어야 한다.
2. 첫 손님은 15초 안에 등장해야 한다.
3. 전체 10단계와 평점 5점은 약 5~8분에 도달하도록 한다.
4. 이 목표는 실제 플레이테스트 전에는 확정하지 말고 조정 가능한 상수로 둔다.
5. 메뉴판 가격을 바꾸지 않고 실제 정산에 밤손님 영업 보너스 ×3를 적용하며, 이 배율을 튜토리얼과 UI에 공개한다.

검증 요구사항:

1. 각 단계에서 잔액 부족, 비용과 같은 잔액, 비용보다 많은 잔액을 확인한다.
2. 버튼 더블 클릭으로 중복 구매되지 않는지 확인한다.
3. 각 단계 구매 직후 실제 좌석·메뉴·속도·직원·방문 간격이 바뀌는지 확인한다.
4. 3단계와 9단계 효과가 저장 복원 때 중복 곱해지지 않는지 확인한다.
5. 구매 후 새로고침하여 돈, 구매 목록, 평점, 직원과 시설이 모두 복원되는지 확인한다.
6. 잘못된 JSON과 과거 version 데이터를 넣어도 시작되는지 확인한다.
7. 달빛 간판만 있는 경우, 평점만 5점인 경우, 둘 다 충족한 경우를 각각 확인한다.
8. npm run build를 실제로 실행하고 결과를 보고한다.

구현 후 생성·수정 파일, 데이터와 시스템 책임, 저장 키/버전, 발견한 오류, 실제 실행한 테스트와 결과를 알려 줘. 실행하지 않은 테스트나 빌드는 성공으로 쓰지 말고 "검증 예정"이라고 적어 줘.
```

## 기대 검증

- 모든 구매는 돈을 정확히 한 번만 차감한다.
- 새로고침 전후 업그레이드 효과와 화면이 같다.
- 잘못된 저장 값이 있어도 게임이 크래시하지 않는다.
- 최종 두 조건이 모두 참일 때만 ResultScene으로 이동한다.
