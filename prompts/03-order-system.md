# 03. 주문·조리·서빙·자동화 프롬프트

## 목적

수동 주문 접수와 음식 운반, 데이터 기반 조리, 결제, 팁, 셰프·서버 자동화를 하나의 끊김 없는 서비스 루프로 연결한다.

## AI에 전달할 프롬프트 전문

```text
현재 "냥포차: 달빛 아래 야식당" 프로젝트에 주문, 조리, 서빙, 결제와 직원 자동화 시스템을 구현해 줘.

기존 손님·좌석 상태 흐름과 프로젝트 구조를 먼저 읽고 그 public API를 사용해. GameScene에 모든 규칙을 넣거나 기존 코드를 중복 구현하지 마. TypeScript strict를 유지하고 any를 사용하지 마.

메뉴 데이터는 다음 값을 단일 기준으로 사용한다.

- fishcake / 어묵 / 기본 조리 3초 / 기본 가격 10냥 / 처음부터 해금
- tteokbokki / 떡볶이 / 기본 조리 5초 / 기본 가격 25냥 / 떡볶이 조리대 구매 후 해금
- fish-bread / 붕어빵 / 기본 조리 7초 / 기본 가격 45냥 / 붕어빵 기계 구매 후 해금

주문 요구사항:

1. ORDERING 상태 손님 머리 위에 주문 접수 아이콘과 Space 안내를 표시한다.
2. 플레이어가 유효 거리 안에서 Space를 누르면 가장 가까운 한 손님의 주문만 접수한다.
3. 현재 해금된 메뉴만 주문 후보가 된다.
4. 토끼, 강아지, 햄스터, 너구리의 선호 메뉴가 해금되어 있으면 선택 확률을 높일 수 있다.
5. 주문에는 고유 id, 손님 id, 메뉴 id, 생성 시각, 접수 시각, 상태를 둔다.
6. 같은 손님의 주문이 중복 접수되지 않게 한다.
7. 접수가 끝나면 손님을 WAITING_FOR_FOOD로 전환하고 조리 대기열에 한 번만 추가한다.

조리 요구사항:

1. CookingSystem은 접수된 주문 대기열과 조리 중 작업, 완성 음식 컬렉션을 관리한다.
2. 메뉴가 해금되지 않았거나 조리대가 없는 주문을 시작하지 않는다.
3. 조리 시간은 메뉴 데이터와 전역 또는 조리대별 속도 배율로 계산한다.
4. 빠른 냄비 업그레이드는 모든 메뉴의 조리 시간을 30% 단축한다. 원 기획의 어묵 전용 효과를 MVP 전역 배율로 조정한 결정이다.
5. 완료된 음식에는 원래 주문 id, 손님 id, 메뉴 id를 유지한다.
6. 조리 완료 시 조리대에 코드 생성 음식 아이콘과 진행 상태를 표시한다.
7. 장면 pause에서는 조리 진행도 멈추고 resume 뒤 이어진다.
8. 장면 shutdown에서는 모든 timer/listener를 정리한다.
9. 조리 완료 알림은 Web Audio API로 합성한 짧은 효과음을 사용한다. 외부 음원 파일을 추가하지 않는다.

수동 운반과 서빙 요구사항:

1. 플레이어는 한 번에 완성 음식 하나만 들 수 있다.
2. 빈손인 플레이어가 완성 음식 가까이에서 Space를 누르면 한 음식만 원자적으로 가져온다.
3. 이미 다른 플레이어/직원이 점유한 음식은 가져오지 않는다.
4. 운반 중인 메뉴를 HUD에 한국어 이름과 코드 생성 아이콘으로 표시한다.
5. 해당 주문의 손님이 WAITING_FOR_FOOD이고 유효 거리 안에 있을 때만 Space로 서빙한다.
6. 다른 메뉴나 다른 주문의 음식을 잘못 전달하지 않는다.
7. 손님이 제거되었거나 주문이 무효가 되면 운반 음식이 게임을 막지 않도록 안전하게 회수 또는 폐기한다.
8. Space 키를 길게 눌렀을 때 한 번의 의도에 여러 상태가 연속으로 지나가지 않도록 just-down 또는 쿨다운 처리를 한다.
9. 서빙 후 손님은 EATING으로 전환하고 메뉴별 또는 공통 식사 타이머를 시작한다.

결제와 만족도 요구사항:

1. EATING 종료 후 PAYING에서 EconomySystem을 통해 한 손님의 정산을 한 번만 처리한다.
2. 메뉴판 가격 10/25/45냥은 유지하되 실제 기본 정산액은 "밤손님 영업 보너스 ×3"를 적용한다. 이 배율은 5~8분 MVP 밸런스 조정이며 튜토리얼과 결제 UI에 숨기지 말고 표시한다.
3. 주문·음식 대기 시간으로 인내심을 계산한다.
4. 매우 만족은 보너스 적용 정산액의 30%, 만족은 10%, 보통 이하에는 0% 팁을 지급한다.
5. 정수 화폐 정책을 사용해 팁 반올림 방식을 한 곳에서 명시한다.
6. 같은 손님의 결제를 두 번 처리하지 않는다.
7. 결제 위치에 코드 생성 동전을 표시하고 플레이어가 닿으면 수거한다. 구현 단순화를 위해 수익 자체를 PAYING 시 반영한다면 시각 동전은 재결제를 일으키지 않는 연출이어야 한다.
8. 수익, 팁, 방문 손님 수, 만족도, 평점을 HUD에 반영한다.
9. 돈과 평점이 바뀔 때 저장 시스템에 안전하게 알릴 수 있는 이벤트를 제공한다.
10. 동전 효과음도 Web Audio API로 합성한다.

직원 자동화 요구사항:

1. 셰프 고양이와 서버 고양이는 외부 이미지 없이 코드 생성 픽셀 텍스처로 표시한다.
2. 셰프는 해당 업그레이드 구매 전에는 비활성 상태다.
3. 셰프가 활성화되면 접수된 주문을 조리 대기열에서 자동 선택해 조리를 시작한다.
4. 서버는 해당 업그레이드 구매 전에는 비활성 상태다.
5. 서버가 활성화되면 완성 음식과 살아 있는 WAITING_FOR_FOOD 손님을 주문 id로 연결해 자동 전달한다.
6. 직원이 가져갈 작업/음식은 선택 즉시 reserved 처리해 플레이어와 다른 직원이 동시에 처리하지 못하게 한다.
7. 직원 이동 중 주문이 무효가 되면 예약을 해제하고 다음 작업으로 넘어간다.
8. 자동화 이후에도 플레이어는 주문 접수, 음식 운반과 서빙을 할 수 있다.
9. 자동화는 돈을 직접 생성하지 않고 반드시 같은 조리·서빙·결제 경로를 사용한다.

시각·사운드 요구사항:

1. 음식, 주문 아이콘, 동전, 하트, 조리 김은 Phaser Graphics/Canvas로 생성한다.
2. 외부 이미지, 이모지 이미지, 음원, 웹폰트 URL을 사용하지 않는다.
3. 손님 입장, 조리 완료, 동전 획득 효과음을 Web Audio oscillator/gain으로 합성한다.
4. 첫 사용자 입력 전 AudioContext 오류가 나지 않게 한다.
5. 음소거 상태를 지원할 수 있는 단일 AudioManager를 사용한다.

검증 요구사항:

1. 세 메뉴의 실제 조리 시간, 메뉴판 가격, ×3 정산액과 팁을 테스트한다.
2. 해금 전 메뉴 주문이 생성되지 않는지 확인한다.
3. 연속 Space 입력으로 주문·픽업·서빙이 한 번에 건너뛰지 않는지 확인한다.
4. 같은 음식이 플레이어와 서버에게 동시에 배정되지 않는지 확인한다.
5. 같은 손님이 두 번 결제하지 않는지 확인한다.
6. 셰프와 서버 구매 전후 동일한 주문이 끝까지 처리되는지 확인한다.
7. npm run build를 실행하고 타입·빌드 결과를 사실대로 보고한다.

구현 후 생성·수정 파일, 시스템별 책임, 발견한 오류, 사람이 판단해야 할 밸런스, 실행한 테스트와 실제 결과를 알려 줘. 실행하지 않은 항목은 "검증 예정"으로 표시해 줘.
```

## 기대 검증

- 해금된 메뉴만 주문되고 조리대·가격과 일치한다.
- 한 음식은 정확히 한 손님에게 한 번만 전달된다.
- 자동화와 수동 조작을 동시에 사용해도 음식·돈이 복제되지 않는다.
- 일시 정지 동안 인내심과 조리·식사 시간이 흐르지 않는다.
