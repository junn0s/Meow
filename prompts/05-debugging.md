# 05. 디버깅·제출 검증 프롬프트

## 목적

기능을 새로 확장하기보다 현재 구현을 읽고 타입, 빌드, 상태 흐름, 저장, 배포 경로와 제출 문서의 불일치를 체계적으로 찾아 수정한다.

## AI에 전달할 프롬프트 전문

```text
"냥포차: 달빛 아래 야식당"의 구현이 끝났다고 가정하지 말고, 현재 저장소를 근거로 해커톤 제출 전 검증과 필요한 최소 수정만 수행해 줘.

중요 원칙:

1. 먼저 git status와 전체 파일 구조를 확인한다.
2. 기존 사용자 변경과 관련 없는 파일을 되돌리거나 대규모로 재작성하지 않는다.
3. 테스트를 실행하지 않고 통과했다고 보고하지 않는다.
4. 오류를 재현한 뒤 원인을 설명하고 가장 작은 안전한 수정으로 해결한다.
5. 타입 오류를 any, @ts-ignore, eslint disable로 숨기지 않는다.
6. 현재 기능 범위를 넘어 온라인, 로그인, 서버, 모바일 조작 같은 새 기능을 추가하지 않는다.
7. 외부 이미지·음원·폰트 패키지를 문제 해결용으로 추가하지 않는다.
8. 문서와 구현이 다르면 어느 쪽이 기획의 단일 기준인지 확인하고 일치시킨다.

1단계 — 정적 확인:

1. package.json의 scripts와 dependency를 확인한다.
2. tsconfig의 strict 설정과 include 범위를 확인한다.
3. Vite base와 에셋 경로가 GitHub Pages 저장소 하위 경로에서 동작하는지 확인한다.
4. 480×270, pixelArt, roundPixels, scale 설정을 확인한다.
5. 외부 URL, PNG/JPG/GIF/SVG/MP3/WAV/OGG/웹폰트 참조를 저장소와 소스에서 검색한다.
6. localStorage key, schema version, parse validation과 초기화 범위를 확인한다.
7. 등록한 keyboard, pointer, resize, timer, scene event가 shutdown 때 정리되는지 확인한다.
8. 실제 파일 구조와 README/게임 설명/AI 보고서의 경로와 기능 설명을 대조한다.

2단계 — 명령 실행:

1. 프로젝트가 요구하는 방식으로 의존성을 설치한다.
2. npm run build를 실행한다.
3. 별도 typecheck script가 있으면 실행한다.
4. 테스트 script가 있으면 실행한다.
5. 실패하면 명령, 핵심 오류 문구, 원인 파일을 기록한다.
6. 수정 후 같은 명령을 다시 실행해 전후 결과를 모두 기록한다.
7. npm run preview를 실행해 production build를 브라우저에서 확인할 준비를 한다.

네트워크 제한이나 권한 때문에 명령을 실행할 수 없다면 우회해서 성공한 척하지 말고 제한과 미검증 범위를 명시한다.

3단계 — 브라우저 기본 흐름:

1. 시작 화면이 10초 안에 보이는지 확인한다.
2. 새 게임과 이어하기 버튼 상태가 저장 유무와 맞는지 확인한다.
3. WASD와 방향키 이동, Space 상호작용, Esc, 마우스 클릭을 확인한다.
4. 대각선 이동이 축 이동보다 과도하게 빠르지 않은지 확인한다.
5. 키를 누른 채 탭 전환 후 돌아왔을 때 이동이 고착되지 않는지 확인한다.
6. 창 크기를 바꿔도 16:9 비율, HUD, 픽셀 선명도가 유지되는지 확인한다.
7. 첫 사용자 입력 전후 AudioContext 콘솔 오류가 없는지 확인한다.

4단계 — 게임 규칙 회귀:

1. 첫 손님이 15초 이내에 등장하는지 측정한다.
2. 좌석보다 많은 손님을 생성해 중복 예약과 대기열 교착을 확인한다.
3. 주문 접수 전후 손님 상태와 조리 대기열이 한 번만 바뀌는지 확인한다.
4. 어묵 3초/10냥, 떡볶이 5초/25냥, 붕어빵 7초/45냥과 실제 밤손님 영업 보너스 ×3 정산을 확인한다.
5. 메뉴 해금 전 해당 주문이 생기지 않는지 확인한다.
6. Space 연타와 길게 누르기로 주문, 픽업, 서빙이 중복되지 않는지 확인한다.
7. 다른 메뉴 음식과 다른 손님 주문이 매칭되지 않는지 확인한다.
8. 한 손님이 결제와 방문 통계를 한 번만 증가시키는지 확인한다.
9. 인내심에 따른 30%, 10%, 0% 팁이 보너스 적용 정산액 기준인지와 정수 반올림을 확인한다.
10. 셰프와 서버 해금 전후 주문이 끝까지 처리되는지 확인한다.
11. 직원과 플레이어가 같은 음식 또는 주문을 동시에 점유하지 않는지 확인한다.
12. pause 중 손님 인내심, 조리, 식사 timer가 멈추는지 확인한다.

5단계 — 업그레이드·저장·클리어:

1. 10단계 이름, 가격, 순서, 효과가 데이터와 UI에서 일치하는지 확인한다.
2. 잔액 부족 버튼 비활성화와 정확한 잔액 구매를 확인한다.
3. 더블 클릭과 연속 입력으로 중복 구매되지 않는지 확인한다.
4. 좌석, 어묵 속도, 메뉴, 직원, 손님 간격 효과가 즉시 반영되는지 확인한다.
5. 구매와 수익 후 새로고침해 돈, 평점, 방문 수, 업그레이드와 시설을 확인한다.
6. 빠른 냄비와 네온 간판 효과가 복원 때 중복 적용되지 않는지 확인한다.
7. localStorage에 잘못된 JSON, 음수 돈, 알 수 없는 id, 과거 version을 넣고 복구를 확인한다.
8. 저장 초기화가 이 프로젝트 키만 삭제하는지 확인한다.
9. 달빛 간판만 구매한 경우에는 클리어되지 않고 평점 5점 안내가 나오는지 확인한다.
10. 평점 5점만 달성한 경우에는 달빛 간판 안내가 나오는지 확인한다.
11. 두 조건을 모두 충족할 때 클리어가 한 번만 실행되고 ResultScene 통계가 맞는지 확인한다.
12. 처음부터 정상 플레이해 첫 업그레이드 1분 이내, 전체 5~8분 목표를 측정한다.

6단계 — 에셋·배포:

1. 브라우저 Network 패널에서 제3자 이미지, 음원, 폰트 요청이 없는지 확인한다.
2. 캐릭터, 음식, 가구, 이펙트가 코드 생성 텍스처인지 확인한다.
3. 효과음이 Web Audio 합성이며 샘플 파일을 읽지 않는지 확인한다.
4. 프로덕션 빌드의 dist를 로컬 하위 경로 조건에서 확인한다.
5. GitHub Pages 실제 URL을 시크릿 창에서 연다.
6. Pages에서 직접 진입, 새로고침, 저장 후 재접속, 음소거를 확인한다.
7. 콘솔 404, MIME, CORS, mixed content 오류가 없는지 확인한다.

7단계 — 보고 형식:

다음 순서로 결과를 작성해 줘.

1. 발견한 문제: 심각도, 재현 방법, 원인, 수정 파일
2. 변경한 파일 전체 목록과 변경 이유
3. 실행한 명령과 종료 코드
4. 자동 검증 결과
5. 수동 브라우저 검증 결과
6. 아직 검증하지 못한 항목과 이유
7. 사람의 판단이 필요한 밸런스·UX
8. README, docs/game-description.md, docs/ai-usage-report.md, docs/asset-licenses.md에 반영해야 할 사실

AI 활용 보고서에는 AI 도구/버전, 프롬프트 전문, 생성·수정 파일, 발생 오류, 사람의 직접 수정, AI 결과를 그대로 사용하지 않은 이유, 테스트 방법, 실제 결과, 최종 반영 여부를 남겨 줘.

실제 빌드와 브라우저 테스트를 하지 못한 항목은 반드시 "검증 예정"으로 표시하고, 성공을 추정하지 마.
```

## 권장 회귀 시나리오

| 시나리오 | 확인할 불변 조건 |
| --- | --- |
| 손님 20명 빠른 생성 | 한 좌석에 활성 예약은 최대 1개 |
| Space 연타 | 주문·음식·결제는 각각 최대 1회 |
| 서버와 플레이어 동시 접근 | 완성 음식 소유자는 한 주체뿐 |
| 저장 직후 새로고침 | 잔액과 효과가 저장 전과 동일 |
| 잘못된 저장 JSON | 크래시 없이 기본 상태 진입 |
| 달빛 간판만 구매 | 평점 5점 전에는 클리어되지 않음 |
| 평점 5점만 달성 | 달빛 간판 전에는 클리어되지 않음 |
| 장면 반복 진입 | timer/listener 수가 누적되지 않음 |
