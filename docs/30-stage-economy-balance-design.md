# 냥포차 30단계 경제·난이도 밸런스 설계서

## 챕터 경제 단위

각 챕터는 같은 원 단위로 되돌아가지 않고 10억 배씩 더 큰 경제 규모에서 시작한다. 배율은 CH.1 `×1`, CH.2 `×10^9`, CH.3 `×10^18`, CH.4 `×10^27`, CH.5 `×10^36`이다. 따라서 첫 메뉴는 각각 `18`, `18B`, `18ab`, `18ae`, `18ah`에서 시작한다. 표기 단위는 `K, M, B, T, aa, ab, ac …`이며 1,000배마다 다음 단위로 진행한다. 이 배율은 메뉴 매출, 30단계 구매비용, 조리대와 현재 챕터에서 새로 사는 상점 아이템에 공통 적용한다.

챕터 이동 시 상점 인벤토리와 장착 상태도 초기화한다. 내부 효과 슬롯은 동일한 밸런스 역할을 유지하지만 이름·설명·색상·시설 아이콘은 달빛 골목, 하와이안 해변, 한옥, 파인다이닝, 오마카세 테마로 전면 교체한다. 시설 효과 강도는 챕터별 `×1.0 / ×1.1 / ×1.2 / ×1.3 / ×1.4`로 올려 반복 구매의 보상을 강화한다.

## 0. 결론부터

기존 10개 일회성 업그레이드를 **30개 성장 단계**로 확장한다. 각 성장 단계에는 돈을 모으는 동안 살 수 있는 세부 구매가 5~6개 있고, 마지막 핵심 구매를 마치면 다음 단계로 넘어간다. 그래서 30단계가 60분짜리 단일 대기 버튼 30개가 되지 않도록 한다.

- 1~5단계는 `30초 → 50초 → 70초 → 90초 → 120초`로 튜토리얼처럼 빠르게 진행한다.
- 6~10단계는 `3:24 → 3:17 → 3:10 → 3:02 → 2:55`로 완만하게 단축한다.
- 11~15단계는 각 5분 36초, 16~20단계는 각 8분 24초, 21~25단계는 각 11분 54초로 기존안의 70%를 적용한다.
- 26~30단계는 각각 14분, 17분 30초, 21분, 24분 30초, 42분으로 맞춘다.
- 기준 총 플레이타임은 **270분 18초, 즉 약 4시간 30분**이다.
- 시간 기준은 광고, 결제, 오프라인 수익을 쓰지 않는 평균 플레이어의 **활성 플레이 P50**이다.
- 후반 난도는 가격만 올려 기다리게 하는 방식이 아니라, 손님 수요·좌석·주방·서빙 병목과 실제 주문 이탈을 함께 올려 만든다.
- 결제 화면의 음식 가격과 실제 기본 결제액을 일치시킨다. 기존의 숨은 `×3` 정산 배율은 제거한다.
- 셰프와 서빙 알바를 boolean 한 명이 아니라 각각 `0~5명`으로 만들고, 실제 동시 처리 슬롯을 늘리게 한다.
- 30단계는 42분짜리 한 번의 대기가 아니라 **8분 24초짜리 달빛 간판 제작 5구간**으로 분할한다.
- 화면은 재생 중인 로파이 한 곡의 실제 길이마다 낮·노을·밤·새벽이 바뀌며, 밤에는 깊은 청색 거리 위에 청록·자홍 네온과 따뜻한 포차 조명이 단계별로 점등된다.

### 요청 구간 해석

최초 요청에는 `15~20은 12분`, `20~25는 17분`처럼 15와 20이 겹쳤다. 최초 기준안에서는 다음처럼 해석했으며, 2026-07-19 완화안은 여기에 70% 배율을 적용한다.

- `11~15`: 각 8분
- `16~20`: 각 12분
- `21~25`: 각 17분

이 해석이 의도와 다르면 구현 전에 시간 배열만 교체하고, 나머지 가격은 같은 공식으로 다시 생성한다.

### 이 문서에서 말하는 “도파민”

신경전달물질을 직접 측정하거나 조작한다는 뜻으로 쓰지 않는다. 이 문서에서는 플레이어가 느끼는 **기대, 진척 확인, 숙련감, 큰 이정표 보상**을 묶어 부르는 일상적 표현으로만 사용한다. 게임 참여 연구는 자율성·유능감·관계성 충족을 즐거움과 연결하지만, 객관적 난이도와 즐거움 사이에 항상 하나의 보편적인 최적점이 존재한다는 결과만 있는 것은 아니다. 따라서 아래 수치는 가설이며 반드시 플레이 데이터로 검증한다. 근거는 [게임 참여와 자기결정성이론 연구](https://journals.sagepub.com/doi/pdf/10.1037/a0019440?download=true), [난이도-기술 균형의 사전등록 반례](https://pmc.ncbi.nlm.nih.gov/articles/PMC9890114/)를 참고한다.

## 1. 레퍼런스에서 확인한 것과 확인하지 못한 것

### 1.1 확인 가능한 구조

공식 스토어 설명에서 Cat Snack Bar는 고양이가 주문을 받고 음식을 만들고 서빙하며, 게임을 꺼도 운영되는 방치형 타이쿤임을 확인할 수 있다. 다만 정확한 경제 산식이나 서버 밸런스 값은 공개하지 않았다. [Google Play 공식 페이지](https://play.google.com/store/apps/details?hl=en&id=com.tree.idle.catsnackbar)

팬이 관리하는 공개 표에서는 다음 구조를 관찰할 수 있다. 아래 수치는 버전 변경이나 표기 오류가 있을 수 있으므로 “공식 내부 공식”으로 취급하지 않는다.

- 조리대는 레벨을 올릴수록 음식 가격이 오르고, Lv.10/25/50/75/100/150/200/250 같은 이정표에서 가격 `×2`와 추가 조리대 보상을 준다. [Worktops 팬 위키](https://catsnackbar.fandom.com/wiki/Worktops)
- 일반 스테이지 표를 역산하면 조리대 비용은 레벨마다 약 `×1.20`, 이정표를 제외한 기본 수익은 약 `×1.08~1.086`에 가까운 구간이 있다. 모든 스테이지의 공식이라고 단정할 수는 없다. [Food Truck](https://catsnackbar.fandom.com/wiki/Food_Truck), [Cafeteria](https://catsnackbar.fandom.com/wiki/Cafeteria)
- 후반 메뉴 예시에서는 다음 메뉴 가격이 이전보다 약 `×37~42` 커지고 조리시간은 5초씩 증가한다. 별도 관리 업그레이드에는 음식 가격의 큰 배수, 조리시간 감소, 손님·직원 추가가 섞여 있다. [Steakhouse 팬 위키](https://catsnackbar.fandom.com/wiki/Steakhouse), [Upgrades 팬 위키](https://catsnackbar.fandom.com/wiki/Upgrades)
- 직원 추가 가격은 단순 선형이 아니라 해당 시점의 수익 자릿수에 맞춰 크게 뛴다. 손님 증가, 셰프·알바 추가, 음식 가격 배수, 속도 향상을 번갈아 배치해 병목을 바꾼다. [Taco Place 팬 위키](https://catsnackbar.fandom.com/wiki/Taco_Place)
- 초기에는 기능을 차례로 소개하고, 특정 음식 홍보나 도우미처럼 짧은 집중 구간을 만든다. [Starting Guide 팬 위키](https://catsnackbar.fandom.com/wiki/Starting_Guide)

Idle 게임 경제의 일반적인 출발점은 `다음 비용 = 기본 비용 × 성장률^보유량`과 같은 지수 비용이며, 오래된 생산원이 완전히 무의미해지지 않도록 이정표 배수를 섞는다. 매끈한 곡선만 계속되면 선택이 사라지기 때문에 진행 속도에 의도적인 굴곡을 만든다는 실무 설명도 있다. [Anthony Pecorella, The Math of Idle Games Part I](https://www.gamedeveloper.com/design/the-math-of-idle-games-part-i), [GDC 2016 발표 자료](https://media.gdcvault.com/gdceurope2016/presentations/Pecorella_Anthony_Quest%20for%20Progress.pdf)

### 1.2 그대로 복제하지 않을 것

Cat Snack Bar의 다음 항목은 정확한 공개 자료가 없거나 이 프로젝트에 그대로 넣으면 부작용이 크다.

- 정확한 손님 생성 간격, 좌석 점유시간, 직원 이동 산식
- 전체 스테이지에 공통인 정확한 비용·수익 공식
- 광고 노출 빈도, 결제 상품, 상자 확률과 의상 합성 경제
- 수만~수억 배 의상 보너스와 운에 좌우되는 영구 성장
- 강제 광고, 접속 강요, 결제로만 피할 수 있는 과도한 그라인드

따라서 이 문서는 Cat Snack Bar에서 관찰한 **완만한 레벨 성장 + 큰 이정표 + 병목 교대 + 자동화 확장**이라는 구조를 참고하되, 실제 수치는 냥포차의 처리량과 요청 시간으로 독립 역산한다.

## 2. 현재 게임에서 바꿔야 할 기준점

현재 코드를 기준으로 한 문제는 다음과 같다.

- 메뉴판 가격은 `10/25/45냥`이지만 실제 결제는 항상 `×3`이라 표시 가격과 경제 기준이 다르다.
- 손님 생성은 기본 3.15초, 네온 간판 이후 2.205초 두 상태뿐이다.
- 좌석은 2개에서 최대 6개, 조리속도는 전역 `×0.7` 한 번만 바뀐다.
- 셰프 업그레이드는 실제 동시 조리 인력을 늘리는 대신 주문 자동 접수에 가깝다.
- 서버는 한 명뿐이고 왕복 약 0.89초, 결제 회수 약 0.65초라 홀 병목이 거의 생기지 않는다.
- 인내심이 0이 되어도 손님이 떠나지 않고, 평점은 사실상 계속 오른다.
- `tipChance`, 메뉴별 최대 주문수량, 여러 개 주문의 추가 조리시간이 충분히 반영되지 않는다.
- 피버, 콤보, 메뉴 레벨, 여러 명의 직원, 실제 실패 페널티가 없다.

관련 기준 파일은 `src/game/systems/EconomySystem.ts`, `src/game/scenes/GameScene.ts`, `src/game/data/menuData.ts`, `src/game/data/upgradeData.ts`, `src/game/entities/Customer.ts`이다.

### 핵심 변경 결정

| 항목 | 현재 | 30단계 설계 |
| --- | --- | --- |
| 성장 | 일회성 업그레이드 10개 | 성장 단계 30개 + 단계당 세부 구매 5~6개 |
| 기본 결제 | 메뉴 가격 `×3` | 표시된 현재 메뉴 가격 `×수량` |
| 메뉴 | 3종 | 6종, 약 5단계마다 1종 해금 |
| 직원 | 셰프/서버 boolean | 셰프 0~5명, 서빙 알바 0~5명 |
| 손님 실패 | 이탈 없음 | 인내심 0이면 이탈·무수익·콤보 초기화 |
| 좌석 | 2~6 | 2~14 |
| 속도 | 전역 1회 | 메뉴 레벨, 전역 주방, 직원 이동을 분리 |
| 보상 구간 | 다음 업그레이드 1개 | 단계당 5~6회 구매와 외형 변화 |
| 최종 조건 | 달빛 간판 + 사실상 자동 상승하는 평점 5 | 최종 시설 완성 후 다음 챕터 버튼 선택 |

평점 5.0을 4.5로 바꾸는 이유는 실제 이탈이 생긴 뒤에도 완벽한 5.0을 요구하면 후반 난도와 클리어 조건이 충돌하기 때문이다. 평점 5.0 표시는 계속 달성 가능하지만 필수 클리어선은 4.5로 둔다.

## 4. 경제 산식

### 4.1 용어

- `R_s`: s단계 시작 시 평균 실효 수익/초
- `T_s`: s단계 목표 시간(초)
- `B_s`: s단계에서 지출할 총 예산
- `c_i`: 단계 안 i번째 세부 구매 가격
- 실효 수익은 실제 성공 주문, 메뉴 믹스, 기대 팁·콤보·피버, 이탈률을 반영한 시뮬레이션 평균이다.
- 광고, 결제, 오프라인 수익은 기준 실효 수익에 포함하지 않는다.

### 4.2 기본 결제

```text
표시 단가 = 메뉴 기본가
          × 메뉴 레벨 배율
          × 메뉴 전용 업그레이드
          × 전체 가격 업그레이드
          (단, 오래된 메뉴는 최신 메뉴 대비 최소 가격을 보장)

소계 = 표시 단가 × 주문수량 × VIP 배율

최종 결제액 = round(
  소계 × 콤보 배율 × 피버 결제 배율
  + 팁
)
```

숨은 `밤손님 영업 보너스 ×3`은 삭제한다. 플레이어가 보는 가격이 기본 결제의 단일 기준이다.

### 4.3 식당 처리량과 실효 수익

```text
도착 처리량 = 1 / 손님 등장 간격

좌석 처리량 = 좌석 수 / 평균 좌석 점유시간

주방 처리량 = 동시 조리 슬롯 수
             / Σ(메뉴 주문비중 × 수량 반영 조리시간)

서빙 처리량 = 서빙 알바 수 / 1회 평균 왕복시간

완료 주문/초 = min(도착, 좌석, 주방, 서빙 처리량)
              × 성공률

실효 수익/초 = 완료 주문/초
              × 평균 객단가
              × 기대 보너스
```

도착률만 높여도 주방이나 서빙이 막히면 수익은 더 늘지 않는다. 이 특성을 이용해 후반 난도를 만든다.

### 4.4 단계 안의 여섯 구매

1~29단계는 총예산을 다음 비율로 나눈다.

```text
구매 가격 비중 f = [6%, 9%, 13%, 17%, 23%, 32%]
구매 후 목표 실효수익 배율 g = [1.10, 1.08, 1.12, 1.10, 1.15, 1.08]
```

여섯 배율을 모두 적용하면 한 단계의 목표 실효 수익은 약 `×1.8178`이 된다. `g`는 UI에 그대로 표시할 능력치가 아니라, 해당 구매가 실제 대기열과 메뉴 믹스까지 반영했을 때 도달해야 할 **실효 수익 목표**다. 직원이나 좌석처럼 병목에 따라 실제 효과가 달라지는 기능을 단순히 `수익 +15%`로 구현하면 안 된다.

순서대로 돈을 모아 구매한다고 가정한 시간 계수는 다음과 같다.

```text
F = 0.06
  + 0.09 / 1.10
  + 0.13 / (1.10×1.08)
  + 0.17 / (1.10×1.08×1.12)
  + 0.23 / (1.10×1.08×1.12×1.10)
  + 0.32 / (1.10×1.08×1.12×1.10×1.15)
  = 0.726275...

B_s = R_s × T_s / F
c_i = nice(B_s × f_i)
```

이 비율이면 단계 내 누적 시간 약 `8% / 20% / 35% / 52% / 74% / 100%` 지점에서 구매가 발생한다. 11분 54초 단계는 대략 1.0, 2.4, 4.2, 6.2, 8.8, 11.9분에 보상을 받는다.

세부 구매의 기본 역할은 다음과 같다. 실제 능력치 상승은 UI에 있는 원시 효과로 표시하고, 괄호의 수치는 시뮬레이션에서 맞출 실효 수익 목표다.

| 슬롯 | 기본 역할 | 구매 후 실효 수익 목표 |
| ---: | --- | ---: |
| 1 | 해금된 모든 메뉴 판매가격 상승 | ×1.10 |
| 2 | 현재 주력 메뉴 조리시간 16% 단축 | ×1.08 |
| 3 | 해금된 모든 메뉴 판매가격 상승 | ×1.12 |
| 4 | 사장·직원 이동속도 5% 상승·직원 작업시간 5% 단축 | ×1.10 |
| 5 | 가게 외형 성장 + 모든 메뉴 판매가격 상승 | ×1.15 |
| 6 | 좌석·메뉴·직원·피버 등 단계 핵심 효과를 숫자로 표시 | ×1.08 |

예를 들어 좌석을 늘렸지만 주방이 이미 포화라면 좌석의 실효 수익은 목표보다 낮다. 이 경우 숨은 수익 보정을 붙이지 않고, 같은 단계의 속도·조리대 세부 구매를 재배치해 목표를 맞춘다.

`nice(x)`는 세 자리 유효숫자로 반올림하는 표시 함수다. `1,000=1K`, `1,000,000=1M`, `1,000,000,000=1B`로 표시하되 내부 계산은 반올림 전 값을 쓴다.

## 6. 메뉴·조리대 세부 규칙

### 6.1 메뉴 기본값

새 메뉴는 이전 메뉴보다 약 35배 비싸고, 조리시간은 3~4초 늘어난다. Cat Snack Bar 후반 표에서 관찰되는 약 40배보다 조금 낮게 잡았다. 메뉴 해금가는 고정 `×50`을 쓰지 않고 4장의 목표시간 공식으로 정한다. 그래야 사용자가 요구한 단계별 시간이 우선 보장된다.

| 메뉴 | 해금 | 기본 표시 단가 | 기본 조리시간 | 이전 메뉴 대비 가격 |
| --- | ---: | ---: | ---: | ---: |
| 어묵 | 시작 | 18 | 4초 | - |
| 떡볶이 | 6단계 | 630 | 7초 | ×35.0 |
| 순대 | 11단계 | 22K | 10초 | ×34.9 |
| 야식 라면 | 16단계 | 790K | 13초 | ×35.9 |
| 달빛 꼬치 | 21단계 | 28M | 16초 | ×35.4 |
| 달빛 정식 | 26단계 | 1B | 20초 | ×35.7 |

신메뉴는 해금 직후 주문 가중치 18%로 시작하고, 해당 5단계 구간이 끝날 때 30%까지 오른다. 이전 메뉴도 빠른 회전과 가격 이정표 때문에 완전히 버려지지 않게 한다.

```text
최종 주문 가중치 예시
1종: [100]
2종: [65, 35]
3종: [45, 25, 30]
4종: [32, 18, 20, 30]
5종: [24, 13, 15, 18, 30]
6종: [18, 10, 12, 13, 17, 30]
```

손님 선호도와 메뉴 홍보 배율을 위 가중치에 곱한 뒤 합이 100%가 되도록 다시 정규화한다.

### 6.2 조리대 레벨

Cat Snack Bar에서 관찰되는 비용 약 `×1.20`, 기본 수익 약 `×1.08` 구조를 그대로 복사하지 않고 다음 범위에서 시작한다.

```text
다음 조리대 레벨 비용
= nice(해당 구간 기준비용 × 1.18^(현재 레벨-1))

메뉴 레벨 가격
= 기본 표시 단가
 × 1.085^(레벨-1)
 × 2^(통과한 가격 이정표 수)
 × 메뉴 전용 배율
 × 전체 가격 배율
```

- 조리대 최대 레벨은 우선 50으로 둔다.
- Lv.10, 25, 50에서 해당 메뉴 가격 `×2`를 준다.
- Lv.10과 50에서는 해당 메뉴 동시 조리대 `+1`을 줄 수 있지만, 전체 셰프 슬롯을 넘을 수 없다.
- 가격 이정표 직전 2레벨에는 UI에서 `다음 ×2`를 항상 미리 보여준다.
- `모든 메뉴 판매가격 상승` 두 번과 `가게 외형 성장 + 메뉴값 상승`은 해금된 모든 메뉴에 각각 가격 레벨 `+1`을 준다.
- `사장·직원 이동속도 5% 상승`은 사장과 직원 이동속도에 실제 `×1.05`, 셰프 주문 접수와 알바 서빙 준비시간에 `×0.95`를 적용한다. 이동속도는 최대 `×1.8`까지 누적된다. 한 단계에서 가격 레벨은 최대 `+3`만 올라 신메뉴의 급등을 막는다.
- 오래된 메뉴의 최저 표시 단가는 최신 메뉴 가격 대비 거리별 `24% / 9% / 4% / 2% / 1%`다.
- 따라서 바로 이전 메뉴도 계속 팔 가치가 있고, 최신 메뉴는 조리시간을 고려해도 주문당 가장 높은 수익을 유지한다.

### 6.3 속도 레벨

가격 레벨과 속도 레벨을 분리한다. 플레이어가 “더 비싸게 팔기”와 “더 빨리 돌리기” 중 현재 병목에 맞는 선택을 할 수 있어야 한다.

```text
메뉴 실제 조리시간
= max(
  기본 조리시간 × 0.35,
  기본 조리시간 × 0.90^메뉴속도레벨 × 전역주방배율
)
```

- 메뉴 속도 1레벨은 16% 단축이며 최저 조리시간은 기본값의 25%다.
- 전역 주방 업그레이드는 3단계 `×0.80`, 17단계 추가 `×0.75`다.
- 최소 시간은 기본 조리시간의 35%다. 무한에 가까운 속도로 내려가지 않는다.

### 6.4 여러 개 주문

```text
수량 반영 조리시간
= 1개 조리시간 × (1 + 0.65 × (주문수량 - 1))
```

2개 주문은 가격 2배·시간 1.65배, 3개 주문은 가격 3배·시간 2.3배다. 대량 주문은 부담인 동시에 처리에 성공하면 효율이 좋은 보상이 된다.

| 단계 구간 | 1개 | 2개 | 3개 | 기대 수량 |
| --- | ---: | ---: | ---: | ---: |
| 1~5 | 100% | 0% | 0% | 1.00 |
| 6~10 | 95% | 5% | 0% | 1.05 |
| 11~15 | 88% | 12% | 0% | 1.12 |
| 16~20 | 78% | 20% | 2% | 1.24 |
| 21~25 | 68% | 26% | 6% | 1.38 |
| 26~30 | 55% | 35% | 10% | 1.55 |

## 7. 셰프·서빙 알바 시스템

### 7.1 역할

- **사장 고양이**: 언제든 주문 접수, 조리 시작, 음식 운반으로 병목을 직접 보조한다.
- **셰프**: 접수된 주문을 실제 주방 슬롯에 배정하고 조리한다.
- **서빙 알바**: 주문 접수, 완성 음식 전달, 돈 수거를 한 번에 한 작업씩 수행한다.

초반에는 플레이어가 직접 일하고, 후반에는 반복 클릭보다 직원 배치와 업그레이드 선택이 중심이 된다. 최종 자동화 목표는 정상 운영의 95% 이상이다.

직원이 해당 단계의 마지막 핵심 구매라면 가격은 다음처럼 단계 경제에 연결한다.

```text
직원 핵심 구매가 = nice(해당 단계 총예산 × 0.32)
                  = nice(R_s × T_s × 0.32 / 0.726275...)
```

따라서 인원수에 고정 배율을 붙이는 것보다 현재 수익 자릿수와 목표 대기시간이 먼저 가격을 결정한다. 첫 직원은 수십 초 안에 체감하고, 후반 직원은 장시간 목표의 핵심 보상이 된다.

### 7.3 서빙 알바 능력과 가격

현재의 약 0.89초짜리 자동 서빙은 제거한다. 실제 위치까지 이동하고 한 명이 한 주문만 담당해야 한다.

| 인원 | 해금 단계 | 핵심 구매가 | 1인 평균 왕복 | 팀 최대 처리량/분 | 이전 고용 대비 명목가 |
| ---: | ---: | ---: | ---: | ---: | ---: |
| 1호 | 7 | 10.0K | 4.2초 | 14.3건 | - |

역할 시간은 상수만 믿지 않고 실제 맵의 평균 이동거리로 측정한다.

```text
역할 1회 시간
= 집기/상호작용 시간
 + 2 × 평균 이동거리 / 이동속도
 + 전달/수거 시간
```

직원을 샀는데 다른 병목 때문에 기대 수입 증가가 3% 미만이면 그 시점에는 판매하지 않는다. 업그레이드 패널에 `예상 처리량 +n%`와 현재 병목을 표시한다.

## 9. 팁·콤보·피버·특수 상황

### 9.1 팁

```text
남은 인내심 75% 이상:
  팁 확률 = min(100%, 손님 기본확률 + 25%p)
  팁 = 소계의 20%

남은 인내심 45~75%:
  팁 확률 = 손님 기본확률
  팁 = 소계의 10%

남은 인내심 45% 미만:
  팁 없음
```

전체 매출에서 팁 기대값은 4~9%, 어떤 구간도 12%를 넘지 않게 한다. 운에 따라 목표시간이 크게 흔들리지 않도록 하기 위함이다.

### 9.2 콤보

| 연속 성공 | 결제 배율 |
| ---: | ---: |
| 0~4 | ×1.00 |
| 5~9 | ×1.05 |
| 10~19 | ×1.10 |
| 20 이상 | ×1.15 |

손님이 이탈하면 콤보는 0으로 돌아간다. 최대 배율은 1.15로 제한해 콤보 실패가 경제를 망치지 않게 한다.

### 9.3 손님 피버

피버는 확률 팝업이나 광고가 아니라, 잘 서빙해 채우고 플레이어가 원하는 순간 누르는 확정 게이지다. 손님만 빨라져 플레이어를 벌주는 기능이 되지 않도록 수요와 공급을 함께 가속한다.

| 레벨 | 해금 | 지속 | 목표 발동 주기 | 손님·조리·서빙 속도 | 결제 | 평균 경제 기여 목표 |
| ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 1 | 9단계 | 18초 | 약 180초 | ×1.40 | ×1.50 | 약 11% |
| 2 | 18단계 | 20초 | 약 210초 | ×1.50 | ×1.60 | 약 13% |
| 3 | 29단계 | 24초 | 약 240초 | ×1.60 | ×1.75 | 약 18% |

- 매우 만족 `+12`, 만족 `+6`, 늦은 서빙 `+2`, 이탈 `-20` 게이지를 적용한다.
- 레벨별 최대 게이지는 100/120/160, 사용 후 최소 잠금은 120/120/150초다.
- 피버의 평균 수익은 5장의 실효수익 표에 이미 포함한다. 피버가 있다고 단계 가격을 추가로 낮추지 않는다.

### 9.4 메뉴 홍보와 VIP

- 23단계부터 선택한 메뉴의 주문 가중치 `×3`, 60초 지속, 180초 재사용 대기를 준다.
- 느린 메뉴를 홍보하면 주방이 막힐 수 있으므로 현재 메뉴별 `가격/조리초`와 예상 병목을 보여준다.
- VIP는 23/25/29단계에서 등장 확률이 5%/8%/12%로 증가한다.
- VIP는 인내심 `×1.25`, 소계 `×2`이며 팁은 중복 지급하지 않는다.

### 9.5 러시

25단계부터 보너스가 아닌 난이도 이벤트로 러시를 넣는다.

| 단계 | 러시 주기 | 지속 | 등장 간격 |

| 25 | 90초 | 25초 | 기본의 ×0.65 |
| 26 | 85초 | 25초 | 기본의 ×0.65 |
| 27 | 80초 | 27초 | 기본의 ×0.62 |
| 28 | 75초 | 27초 | 기본의 ×0.60 |
| 29 | 70초 | 30초 | 기본의 ×0.58 |

러시 10초 전 화면과 소리로 예고한다. 피버를 러시에 맞춰 쓸지, 평시에 수익용으로 쓸지 선택하게 한다.

## 10. 평점·오프라인·진행 안전장치

### 10.1 평점

최근 서비스 결과를 지수가중평균으로 계산한다.

```text
서비스 점수:
매우 만족 1.00 / 만족 0.85 / 늦음 0.60 / 이탈 0.00

평균 = 이전평균 × 0.92 + 새점수 × 0.08
표시 평점 = 1 + 4 × 평균
```

30단계 최종 시설을 완성하면 평점과 관계없이 챕터 결과 화면으로 이동한다. 다음 가게는 자동 전환하지 않고 사용자가 `다음 챕터` 버튼을 눌렀을 때만 시작한다.

### 10.2 오프라인 수익

오프라인 수익은 10단계 이후 해금하되 기준 목표시간에서는 제외한다.

```text
기준 RPS = 종료 전 5분의 피버·VIP 제외 중앙 RPS
원시 오프라인 수익 = min(오프라인 시간, 2시간) × 기준 RPS × 0.50
실제 지급 = min(원시 오프라인 수익, 다음 단계 총예산의 35%)
```

한 번 복귀해 다음 단계를 통째로 건너뛰지 않고, 보통 세부 구매 1~2개를 살 수 있는 수준이다. 초기 버전에는 광고 2배를 넣지 않는다.

### 10.3 진행불능 방지

- 수익이 0인 상태가 20초 지속되면 대기열·직원 상태를 진단하고 안전하게 재할당한다.
- 저장 시 현재 단계, 세부 구매, 메뉴·속도 레벨, 직원 작업, 피버 게이지를 버전과 함께 기록한다.
- 금액은 1조보다 커져도 안전한 범위지만 표시와 저장을 한 곳에서 처리한다.
- 숨은 난이도 보정으로 손님 속도를 실시간 변경하지 않는다.
- 접속 예약 보상, 놓치면 손해 보는 연속 출석, 강제 광고는 넣지 않는다. Idle 게임의 윤리적 설계에서 진행불능과 강제 일정의 위험을 지적한 연구를 참고한다. [CHI idle-game 설계 연구](https://par.nsf.gov/servlets/purl/10174274), [게임 다크패턴 분류](https://www.fdg2013.org/program/papers/paper06_zagal_etal.pdf)

## 11. 지루하지 않게 만드는 보상 리듬

### 11.1 작은 보상과 큰 보상을 분리한다

- 각 단계의 8/20/35/52/74/100% 지점에 세부 구매를 둔다.
- 가격·속도 레벨은 작은 상승, Lv.10/25/50과 5단계 끝은 큰 가격 배수와 외형 변화를 준다.
- 새로운 메뉴는 6/11/16/21/26단계 시작에 배치해 직전 보스 구간 뒤에 즉시 새로운 그림과 전략을 보여준다.
- 직원·좌석·주방 중 병목을 번갈아 바꿔 “항상 가장 최신 메뉴만 누르기”가 정답이 되지 않게 한다.

### 11.2 5단계 톱니형 곡선

각 5단계 구간은 `새 콘텐츠 → 수요 증가 → 병목 → 자동화/해결 → 큰 가격 배수` 순서로 간다. 5/10/15/20/25단계는 강한 이정표이고, 다음 구간 첫 단계는 신메뉴로 잠깐 압력을 해소한다. 전체 목표시간은 길어지지만 순간 체감은 계속 한 방향으로 답답해지지 않는다.

### 11.3 최종 42분

30단계는 다음 다섯 구간으로 보인다.

1. 8분 24초: 간판 틀 설치
2. 16분 48초: 달 모양 판 부착
3. 25분 12초: 등불 점등 준비
4. 33분 36초: 고양이 문양 완성
5. 42분: 전체 간판 점등과 클리어

각 구간에는 `×1.12` 수익과 외형·대사가 있다. 확률 상자는 쓰지 않는다. 어려움은 긴 총합과 높은 운영 압력에서 만들고, 무반응 대기에서 만들지 않는다.

## 12. 시뮬레이션과 합격 기준

가격표는 수학적 시작점일 뿐이다. 실제 게임은 이동거리와 플레이어 반응 때문에 표와 달라지므로 데이터 기반 이산 이벤트 시뮬레이터를 먼저 만든다.

### 12.1 플레이어 프로필

| 프로필 | 반응 | 이동·선택 | 피버 활용 |
| --- | ---: | --- | ---: |
| 숙련 | 0.45초 | 최적에 가까움 | 95% |
| 평균 | 0.90초 | 보통 오버헤드 | 75% |
| 초보 | 1.60초 | 이동 25% 비효율 | 50% |

단계·프로필마다 최소 1,000개 고정 seed를 실행한다.

### 12.2 필수 로그

- 단계 완료시간 P50/P75/P90
- 메뉴별 주문수, 매출, 가격/조리초
- 도착·좌석·주방·서빙 처리량과 이용률
- 평균·최대 대기시간, 이탈률
- 팁·콤보·VIP·피버의 매출 기여율
- 각 구매 전후 실효 수익 증가율
- 잔액과 다음 구매까지 무행동 시간

### 12.3 합격선

- 1~5단계는 초보 P75가 각각 2분 이내다.
- 6단계 이후 평균 P50은 목표시간 `±10%`다.
- 모든 단계에서 `P90 / P50 < 1.5`다.
- 숙련자는 평균보다 20~30% 빠르고, 초보는 후반 평균보다 25~40% 느린 범위다.
- 어떤 seed에서도 영구 수익 정지, 무한 대기열, 유실 주문이 없어야 한다.
- 한 메뉴나 업그레이드가 모든 단계에서 항상 최적이면 이정표·가중치·가격을 재배치한다.
- 평시 병목 이용률 0.95 초과가 30초 이상 이어지지 않는다.
- 긴 단계에서도 다음 구매나 확정 외형 보상까지 16분을 넘지 않는다.

### 12.4 반복 보정

시뮬레이션 중앙시간이 목표와 다르면 가격을 이분탐색하는 것이 우선이다. 빠른 1차 보정은 다음 감쇠식을 쓸 수 있다.

```text
새 가격
= 기존 가격
 × (목표 중앙시간 / 실제 중앙시간)^0.65
```

가격 보정 뒤에는 등장 간격과 조리·서빙 병목을 따로 확인한다. 가격만 높여서 난도를 맞추지 않는다.

## 13. 시각 디자인 조사와 계획

### 13.1 조사에서 가져올 핵심

대한민국 공식 문화 소개는 포장마차를 주로 밤에 운영되는 작은 천막·임시 구조의 길거리 음식점이자, 따뜻한 음식과 일상의 감정·공동체가 만나는 공간으로 설명한다. 서울시의 야시장 자료에서도 어두운 한강과 도시를 배경으로 반짝이는 야간 조명이 핵심 장면으로 나타난다. [Korea.net 포장마차 문화 소개](https://www.korea.net/K-InfoHub/SubMainDetail/view?articleId=4096&headwordCd=74&headwordGroupCd=28&pageIndex=1), [서울시 밤도깨비 야시장](https://english.seoul.go.kr/seoul-bamdokkaebi-night-market/)

실제 야간 포차 사진에서 반복되는 시각 요소는 다음과 같다. [서울 야간 식당 사진 레퍼런스](https://unsplash.com/photos/a-group-of-people-sitting-outside-of-a-restaurant-at-night-jelYkM9qpTk)

- 주황·빨강 천막과 투명 비닐 커튼
- 작고 가까이 놓인 플라스틱 의자와 테이블
- 어두운 거리와 밝은 내부의 강한 명암 대비
- 증기, 빗물, 젖은 바닥의 길고 끊긴 반사
- 손글씨 메뉴판, 작은 전구, 한글 네온사인
- 차가운 바깥색과 따뜻한 음식·사람색의 공존

레퍼런스 이미지는 분위기와 구조를 분석하는 데만 사용한다. 사진이나 다른 게임의 픽셀 에셋을 복사하지 않고 모든 그래픽은 지금처럼 Phaser Graphics/Canvas로 직접 생성한다.

### 13.2 아트 디렉션: “푸른 도시 안의 따뜻한 달빛 포차”

밤 화면을 전부 파랗게 덮으면 음식과 캐릭터까지 죽는다. 최종 방향은 다음의 세 겹 대비다.

1. **바깥 도시**: 남색·군청·청회색으로 차갑고 깊게 만든다.
2. **포차 내부**: 호박색·크림색 조명으로 음식, 고양이, 증기를 따뜻하게 살린다.
3. **성장 네온**: 청록을 주색, 전기 파랑을 보조색, 자홍을 10~15% 포인트로 사용한다.

낮에는 천막의 재질, 나무, 철판, 손글씨 메뉴처럼 “작지만 성실한 가게”가 잘 보인다. 밤에는 같은 구조 위에서 전구·간판·반사·증기·도시 불빛이 켜져, 플레이어가 키운 가게가 낮보다 훨씬 화려하게 보인다.

### 13.3 낮과 밤의 시간 규칙

실제 시계나 사용자의 시간대에 연결하지 않는다. 실제 밤에만 멋진 화면을 볼 수 있게 하면 테스트와 플레이 경험이 불공평해지기 때문이다.

```text
낮1→노을1→밤1→새벽1 / 낮2→노을2→밤2→새벽2
낮1→노을1→밤3→새벽1 / 낮2→노을2→밤1→새벽2
낮1→노을1→밤2→새벽1 / 낮2→노을2→밤3→새벽2

각 시간대 길이 = 해당 배경음악 파일의 실제 재생 길이
```

- 메뉴·일시정지·튜토리얼 중에는 시계가 멈춘다.
- 오프라인 시간은 수익에는 반영되지만 화면 시간을 건너뛰지 않는다.
- `worldClockMs`를 저장해 새로고침 뒤 같은 시간대에서 이어간다.
- 첫 밤은 낮1과 노을1이 끝나는 약 4분 53초에 시작한다.
- 1차 구현에서는 낮과 밤이 경제 수치에 영향을 주지 않는다. 야간 수익 보너스를 넣으면 5장의 목표시간이 흔들리므로 디자인 시스템을 먼저 안정화한다.

시간 전환은 8초 동안 두 팔레트를 교차 페이드한다. 밤 진입 순간에는 1.2초 동안 `천막 전구 → 메뉴 네온 → 달빛 간판` 순서로 켜진다. 지속적인 빠른 깜빡임은 사용하지 않는다.

### 13.5 화면 레이어와 기술 구조

현재 `createGameBackdrop()`은 고정된 밤 배경 하나를 그리고 간판 텍스처만 교체한다. 이를 상태를 가진 시각 시스템으로 바꾼다.

```text
DayNightController
 ├─ worldClockMs / phase / phaseProgress
 └─ phaseChanged 이벤트

DinerVisualSystem
 ├─ BackdropRenderer: 하늘·건물·거리의 시간대 텍스처
 ├─ DecorProgression: 30단계 시설·장식 표시
 ├─ LightRig: 전구·네온·림라이트 풀
 ├─ ReflectionLayer: 젖은 바닥의 1~2px 반사
 └─ WeatherLayer: 비·증기·새벽 안개
```

- 게임 월드는 기존처럼 왼쪽 `350×270`, 오른쪽 `130×270`은 HUD·업그레이드 패널로 유지한다.
- `VisualPhase = day | sunset | night | dawn`, `VisualTier = 0~6`을 데이터로 둔다.
- 각 시간대·티어 배경은 미리 생성한 텍스처를 교차 페이드한다. 매 프레임 Graphics 전체를 다시 그리지 않는다.
- 조명은 블러 셰이더 대신 저해상도 원·사각형을 `ADD` 블렌드와 낮은 alpha로 겹쳐 픽셀 스타일을 유지한다.
- 월드와 UI를 별도 Container 또는 Camera로 분리해 야간 색조가 글자와 버튼까지 물들이지 않게 한다.
- 픽셀아트는 NEAREST와 정수 좌표를 지키고, 반사도 1~2픽셀 수평 띠로 끊어서 표현한다.

#### 5단계별 전체 실루엣

- **1~5 낡은 낮 포차**: 손수레, 붉은 천막, 전구 한 개. 밤은 따뜻하지만 아직 소박하다.
- **6~10 동네 인기 포차**: 천막이 넓어지고 첫 청록 네온이 생긴다.
- **11~15 비 오는 야시장**: 투명 비닐, 찜솥 증기등, 반사가 늘어나 화면 깊이가 생긴다.
- **16~20 심야 라면바**: 스테인리스 주방, 푸른 불꽃, 큰 증기가 중심이다.
- **21~25 달빛 야시장**: 야외 좌석·배너·청록/자홍 조명이 화면 바깥까지 확장된다.
- **26~30 달빛 야식당**: 달 모양 네온, 다중 서비스 라인, 완성 간판으로 가장 화려한 밤을 만든다.

### 13.7 캐릭터·음식·UI 디자인

#### 캐릭터

- 낮에는 기존 고유색을 유지하고, 밤에는 바깥쪽 1픽셀에 청색 림라이트, 포차 쪽 면에 주황 하이라이트를 준다.
- 셰프와 알바 수가 늘 때 같은 스프라이트를 겹치지 않고 모자띠·앞치마·트레이 색을 달리한다.
- 직원 등급은 색만으로 구분하지 않고 모자·배지·도구 실루엣도 함께 바꾼다.
- 손님 인내심이 낮아질 때 표정·자세·게이지가 함께 변한다.

#### 음식

- 음식은 밤에도 파랗게 틴트하지 않는다. 따뜻한 고유색과 1픽셀 밝은 테두리를 유지한다.
- 신메뉴 해금 시 0.8초 확대, 증기, 작은 가격표 뒤집기 연출을 사용한다.
- 고가 메뉴일수록 그릇 크기보다 증기·곁들임·접시 무늬가 늘어난다.

#### UI

- HUD에 해/노을/달 아이콘과 다음 시간대까지의 작은 진행 링을 둔다.
- 업그레이드 패널은 시간대에 따라 배경색을 바꾸지 않는다. 네온 장면에서도 글자 대비를 유지한다.
- 구매 가능은 색 외에 동전 아이콘·테두리·텍스트를 함께 바꾼다.
- 피버는 청록, 러시는 자홍/빨강을 쓰되 아이콘 모양과 라벨도 달리한다.

### 13.8 움직임·접근성·성능

- 네온은 밤 진입과 해금 순간에만 짧게 켜지고 상시 깜빡이지 않는다.
- 번쩍임은 초당 3회를 넘기지 않는다.
- `prefers-reduced-motion` 또는 게임 설정이 켜지면 비·증기 수를 60% 줄이고, 네온 순차 점등을 0.3초 페이드로 바꾼다.
- 일반 크기 UI 글자는 배경과 최소 4.5:1, 아이콘·게이지는 3:1 대비를 목표로 한다. 색 하나만으로 상태를 전달하지 않는다. [W3C WCAG 2.2](https://www.w3.org/TR/WCAG22/)
- 960×540 실제 버퍼와 480×270 논리 좌표를 유지한다.
- 동적 광원 24개, 화면 파티클 60개를 1차 상한으로 두고 데스크톱 브라우저 60fps를 목표로 한다.
- 터치 기기의 기본 균형 모드는 글자 해상도를 유지하면서 30fps 상한과 `low-power` GPU 힌트를 사용하고, 네온 16개·비 24개·안개 6개까지만 표시한다. 사용자는 고화질 60fps 또는 반사·다수 연출을 끄는 24fps 절전 모드를 선택할 수 있다.

### 13.9 디자인 완료 기준

- 1/10/20/30단계를 낮·노을·밤으로 캡처한 12장 기준 스크린샷을 만든다.
- 모든 스크린샷에서 캐릭터, 주문 아이콘, 조리 완료 음식, 다음 업그레이드가 1초 안에 구분된다.
- 30단계 밤은 1단계 밤보다 네온 계층, 반사색, 움직이는 조명, 좌석·시설 실루엣이 명확히 많아야 한다.
- 낮과 밤이 같은 가게의 두 상태로 보이고, 완전히 다른 맵처럼 순간이동하지 않는다.
- 시간 전환 중 버튼 위치·물리 충돌·게임 속도·경제 수치는 변하지 않는다.
- 축소 화면과 reduced-motion에서도 진행 정보가 손실되지 않는다.
- 각 단계 해금 직후 외형 변화가 저장되고 새로고침 뒤 동일하게 복원된다.

## 14. 이 문서를 구현하는 1단계부터 10단계

> 앞으로 사용자가 **“1단계”**라고만 말하면 아래 **구현 로드맵 1단계**를 즉시 시작한다. 게임 안의 성장 1단계를 뜻하지 않는다. 한 단계의 완료 기준을 통과한 뒤 결과를 보고하고, 다음 번호를 기다린다.

### 구현 현황 (2026-07-16)

- **1단계 완료**: 30단계·6메뉴·8직원 데이터, 낮/노을/밤/새벽 토큰, 6티어, `SaveData v3`, v1/v2→v3 자동 마이그레이션
- **2단계 완료**: 고정 seed·3프로필·단계별 1,000회 시뮬레이션, P50/P90·성공률·병목 리포트, 12장 시각 검증 행렬
- **3단계 완료**: 표시가=기본 결제가, 통합 결제식, K/M/B 표기, 성장 수식, 음악 길이 기반 활성 시계, 4개 시간대 교차 페이드, HUD 연동
- **4단계 완료**: 30단계 세부 구매 런타임, 챕터·구매 진행 UI, 14석 확장, 6티어 천막·간판·네온·달빛 아치, 단계/외형 저장 복원
- **5단계 완료**: 6메뉴·6조리대, 해금/주문 가중치/다량 주문, 메뉴별 가격·속도 레벨, 음식·주문 아이콘과 프리미엄 조리대 아트
- **6단계 완료**: 단계별 등장 간격·인내심·수량, 무결제 이탈·최근 서비스 평점·연속 이탈 안전장치, 종족별 성향·VIP·25~30단계 러시, 야간 림라이트와 상태 기호
- **7단계 완료**: 셰프·서버 각 5명의 독립 작업 점유, 실제 동시 조리 슬롯, 정확한 주문자 왕복 배달, 취소 티켓 회수, 5색 성장 유니폼과 직원 작업등
- **8단계 완료**: 확정 게이지 FEVER Lv.1~3, 30초 메뉴 홍보, 다음 구매비 45% 상한 오프라인 수익, 24광원·60파티클 풀, 네온 순차 점등·반사·안개·reduced-motion
- **9단계 부분 완료**: 단계별 가격 이분탐색, 1,000 seed × 3프로필 합격선, FEVER 10~17% 기여율, 대비·터치 크기·모션·렌더 상한 정적 감사 완료. 신규 실브라우저 E2E와 12장 재촬영은 실행 환경 사용량 제한으로 보류
- **10단계 부분 완료**: 달빛 간판 5부품 순차 점등, 시간대 합성 앰비언스, 4단계 튜토리얼, 모바일 조작, README·게임 설명서·AI 문서·Sites 빌드 최신화. GitHub Pages Actions 배포와 공개 URL HTTPS 200 응답 확인 완료
- 검증: `typecheck`, 프로덕션/Sites `build`, 기초 스모크 테스트, 30단계 밸런스 시뮬레이션, 릴리스 정적 감사 통과. 1~6단계 시점의 실브라우저 1/8/16/24/30단계·저장 복원·VIP/러시·콘솔 오류 0건 기록은 유지하되 7~10단계 신규 검증으로 간주하지 않음

### 1단계 — 데이터 모델·저장·디자인 토큰 고정

- `StageConfig`, `EconomyConfig`, `MenuProgress`, `WorkerProgress`, `FeverState` 타입 추가
- `VisualPhase`, `VisualTier`, `WorldVisualState`, `worldClockMs` 타입과 저장 필드 추가
- 30단계·메뉴·직원·난이도 seed 데이터를 코드 데이터로 옮김
- 낮/노을/밤/새벽의 의미 기반 팔레트 토큰과 6개 포차 티어 정의
- 기존 저장을 새 저장 구조로 옮기는 v1/v2→v3 마이그레이션 작성
- 아직 실제 게임 규칙과 화면은 바꾸지 않음
- 완료 기준: `typecheck`, `build`, 구 저장 복원, 시간대·티어 직렬화 테스트 통과

### 2단계 — 밸런스 시뮬레이터와 시각 검증 틀

- Phaser 화면 없이 손님·좌석·주방·서빙·수익을 계산하는 이산 이벤트 시뮬레이터 작성
- 평균/초보/숙련 프로필과 고정 seed 지원
- 30단계 P50/P90, 병목, 실패율 보고서 출력
- 1/10/20/30단계 × 낮/노을/밤의 12장 기준 스크린샷 규격과 파일명을 고정
- 시간대별 필요한 장식·광원 목록을 데이터 스냅샷으로 검증
- 완료 기준: 같은 seed 재현, 1,000회 실행, 결과 JSON/표, 12장 시각 테스트 매트릭스 생성

### 3단계 — 결제 경제와 낮·밤 기반 화면

- 숨은 `×3` 제거, 표시 가격과 결제 기준 통일
- 팁·수량·콤보·VIP·피버를 합성하는 단일 결제 함수 작성
- K/M/B 포맷터와 조리대 가격·수익 이정표 구현
- `DayNightController`와 6개 음악 루프 기반 활성 플레이 시계 구현
- 고정 밤 배경을 낮/노을/밤/새벽 텍스처 교차 페이드 구조로 교체
- 월드와 UI 색조 레이어를 분리하고 HUD에 시간대 아이콘 추가
- 완료 기준: 결제 단위 테스트, 새로고침 시간 복원, 네 시간대 강제 전환, UI 대비 유지

### 4단계 — 30단계 진행 UI와 포차 6티어 외형

- 기존 10단계를 30단계 + 세부 구매 구조로 교체
- 단계 진행률, 다음 구매, 5단계 챕터, 30단계 5부품 표시
- `DecorProgression`으로 1~30단계의 천막·좌석·간판·소품 변화를 연결
- 1~5/6~10/11~15/16~20/21~25/26~30의 전체 실루엣 6종 구현
- 구매 직후 낮 외형과 밤 조명이 동시에 저장·복원되게 함
- 완료 기준: 디버그 자금으로 1→30 순회, 각 단계 외형 변화 1개 이상, 저장 복원 일치

### 5단계 — 6개 메뉴·조리대와 음식 아트

- 어묵·떡볶이·순대·라면·꼬치·달빛 정식 구현
- 메뉴 주문 가중치, 선호, 여러 개 주문 조리시간 반영
- 가격 레벨과 속도 레벨 UI 분리
- 메뉴마다 낮의 재질 표현과 밤의 따뜻한 1픽셀 하이라이트 제작
- 6개 조리대, 음식 아이콘, 해금 확대·증기·가격표 연출 추가
- 완료 기준: 잠긴 메뉴 주문 없음, 6종 공식 테스트, 낮/밤 모두 음식 식별 가능

### 6단계 — 손님 난이도·평점과 야간 가독성

- 30단계 등장 간격·인내심·수량 확률 적용
- 인내심 0 이탈, 안전장치, 최근 서비스 평점 구현
- 토끼·햄스터·강아지·너구리 차이와 VIP·러시 구현
- 캐릭터에 밤 청색 림라이트·포차 방향 주황 하이라이트 적용
- 표정·자세·게이지를 함께 바꿔 색 없이도 인내심 상태를 알 수 있게 함
- 완료 기준: 단계별 이용률·이탈률 범위 충족, 모든 시간대에서 손님·주문 아이콘 1초 내 식별

### 7단계 — 여러 직원과 성장하는 유니폼

- boolean 직원을 인원수와 독립 작업 상태로 교체
- 실제 셰프 동시 조리 슬롯과 서버 왕복 이동 구현
- 4명까지 고용, 가격·속도·작업 전환 반영
- 셰프 모자띠·앞치마, 알바 트레이·배지로 직원별 실루엣 구분
- 직원 수에 따라 주방 작업등·서비스 유도등이 하나씩 늘어나는 연출 연결
- 완료 기준: 주문 중복·유실 0, 처리량 오차 5% 이내, 낮/밤 직원 4명 시각 구분

### 8단계 — 피버·네온·반사·날씨·오프라인

- 확정 게이지 피버 Lv.1~3, 메뉴 홍보, 14석, 오프라인 수익 구현
- `LightRig`, `ReflectionLayer`, `WeatherLayer`를 풀링 방식으로 구현
- 피버 청록 점등과 러시 자홍 경고를 모양·문구와 함께 구분
- 밤 진입 1.2초 네온 순차 점등, 빗물 1~2픽셀 반사, 새벽 안개 추가
- reduced-motion에서 비·증기 감소와 짧은 페이드 적용
- 완료 기준: 피버 매출 10~18%, 오프라인 단계 스킵 불가, 동적 광원 24·파티클 60 상한 준수

### 9단계 — 자동 밸런싱·시각 회귀·실브라우저 테스트

- 시뮬레이터로 각 단계 가격 이분탐색, 1,000 seed × 3프로필 합격선 검사
- 실제 브라우저에서 신규/복원/중간/최종 단계 E2E 수행
- 12장 기준 스크린샷과 현재 화면의 시각 회귀 비교
- UI 글자 4.5:1, 아이콘 3:1 대비, 번쩍임·reduced-motion·작은 화면 검사
- 30단계 밤이 1단계 밤보다 분명히 화려한지 네온·반사·실루엣 체크리스트로 검증
- 완료 기준: 12절의 밸런스 합격선, 13.9의 디자인 합격선, 콘솔 오류 0, 목표 브라우저 60fps

### 10단계 — 최종 연출·문서·배포 검증

- 5단계별 외형 변화와 달빛 간판 5부품 점등·클리어 연출 마감
- 낮→노을→밤의 사운드 앰비언스와 화면 전환 타이밍 정리
- 튜토리얼, 접근성, 모바일/작은 화면, 사운드 설정 점검
- README·게임 설명서·AI 활용 문서·밸런스/디자인 문서 최신화
- 프로덕션 빌드, Sites/GitHub Pages 실제 URL, 저장 마이그레이션 검증
- 완료 기준: 설치 없이 URL 플레이, 모든 시간대와 새 게임→최종 클리어에 치명적 오류 0

## 16. 최종 결정 요약

이 설계는 Cat Snack Bar의 비공개 공식을 복제한 것이 아니다. 공개 표에서 관찰되는 `지수 비용`, `완만한 기본 수익`, `큰 가격 이정표`, `직원·손님·속도 병목 교대`, `짧은 부스트` 구조를 가져와 냥포차의 실제 처리량과 사용자가 지정한 시간으로 다시 계산한 독자 밸런스다.

구현 중 임의로 가격을 손대지 않는다. 모든 변경은 `목표시간 → 처리량 시뮬레이션 → P50/P90 검증 → 가격 보정` 순서로 수행한다. 특히 후반의 “매우 어려움”은 장시간 무대기나 강제 광고가 아니라, 높은 이용률·다량 주문·이탈·러시 속에서 어떤 병목을 먼저 풀지 선택하는 운영 난도로 만든다.

시각적으로는 `낮의 소박하고 재질감 있는 포차 → 노을의 첫 점등 → 푸른 도시 속 따뜻하고 화려한 네온 포차`가 반복되는 하루를 만든다. 30단계의 모든 경제 구매는 최소 하나의 외형 보상과 연결하고, 최종 밤은 청록·파랑·금색 달빛 간판이 완전히 켜지는 장면으로 마무리한다.
